What this CSV is

alex_event_ledger_ref9_plus_targets5.csv is a deterministic spreadsheet view of the M&A bidding process for 14 deals: 9 reference deals (verified against their SEC filings) and 5 target deals (extracted but pre-verification). It is generated by

python scripts/export_alex_event_ledger.py --scope all \
    --output output/review_csv/alex_event_ledger_ref9_plus_targets5.csv

which reads the latest trusted graph snapshots in output/extractions/{slug}.json and projects them through pipeline/deal_graph/project_alex_event_ledger.py. There is no LLM call in this step. Re-running the command on unchanged snapshots produces a byte-identical CSV.

What one row represents

Each row is one event, one relation, or one participation count. It is not:

  • one row per bidder (consortiums are preserved as single bidders)
  • one row per claim (multiple claims may collapse into one canonical event)
  • one row per filing page (a filing page typically contributes many rows)
  • a price-projection ledger (no bI, bF, T, admitted — those fields are reserved for downstream econometric tooling and are not in this file)

Within each deal, rows are sorted by event_date, then by a fixed event-priority map (advisor first, contacts, NDAs, IOIs, bids, advancements, dropouts, exclusivity, financing commitments, merger execution; participation counts come last). The event_order column encodes that order in multiples of 10.

Reading a single row end-to-end

To anchor the column-by-column reference that follows, here is one row read in full. We pick providence-worcester:40, which is a typical bid submission with all the bid-specific fields populated.

deal_slug=providence-worcester  TargetName=PROVIDENCE & WORCESTER RR CO  Acquirer=GENESEE & WYOMING INC  DateAnnounced=2016-08-15
event_order=40  event_date=2016-07-12  date_precision=exact
row_unit=bidder  party_name=Party C  bidder_class=strategic
initiated_by=bidder  event_family=bid  event_code=bid_submitted  stage=initial  formality=informal
dropout_side=not_applicable  dropout_reason=not_applicable
bid_value=21.0  bid_value_lower=  bid_value_upper=  bid_value_unit=per_share  consideration_type=unspecified
review_label="Party C submitted initial bid (21.0)"
evidence_page=36
evidence_quote_full="On July 12, 2016, Party C submitted an IOI with an offer price per share of $21.00"
confidence=high  needs_human_attention=0
source_claim_ids=claim_da42b5669b149876

Walking left to right:

  • deal_slug = providence-worcester. The lowercase identifier of the deal; matches the directory under data/filings/.
  • TargetName / Acquirer / DateAnnounced. Pulled from data/filings/{slug}/manifest.json via state/progress.json. Same for every row of a given deal.
  • event_order = 40. The 4th row of this deal in chronological / priority order. Multiples of 10 leave room for inserts on review without reflowing the whole sheet.
  • event_date = 2016-07-12, date_precision = exact. The filing states a precise date; the date column is in ISO YYYY-MM-DD form.
  • row_unit = bidder. The party_name is a single named/anonymised bidder, not a count or a deal-level placeholder.
  • party_name = Party C. The filing's anonymous handle for this bidder.
  • bidder_class = strategic. The selected LLM backend emitted (or downstream canonicalisation derived) a strategic class.
  • initiated_by = bidder. The evidence quote contains "submitted", which is one of the bidder-verb keywords.
  • event_family = bid, event_code = bid_submitted, stage = initial, formality = informal. The IOI is treated as a first-round-bid event; first-round bids are informal by convention in the projection map.
  • bid_value = 21.0, bid_value_unit = per_share, consideration_type = unspecified. The filing supports a $21.00 per-share figure but does not state cash-vs-stock here, hence unspecified consideration.
  • evidence_page = 36. The filing page where the quote was bound.
  • evidence_quote_full = "On July 12, 2016, Party C submitted an IOI with an offer price per share of $21.00". The byte-exact filing substring. No paraphrasing.
  • confidence = high. The selected LLM backend marked the underlying claim high-confidence; the quote tightly supports every typed field of the claim.
  • needs_human_attention = 0. The row is clean; nothing flagged.
  • source_claim_ids = claim_da42b5669b149876. The deterministic ID of the claim that produced this row. Searchable in output/extractions/providence-worcester.json and the audit deal_graph_v2.json.

How to spot anomalies

Recommended review heuristics, ordered by leverage:

  1. needs_human_attention = 1. Sort the CSV on this column descending. Anything flagged is the first place to look. (In the current snapshot: empty.)
  2. confidence = medium or low. Filter and read every quote. The 23 medium rows in this CSV are listed in the confidence reference above.
  3. dropout_reason ∉ {unknown, not_applicable}. The keyword detector is approximate. Any non-trivial dropout reason should be confirmed against the quote. Particularly: valuation and financing are the easiest false positives (any quote mentioning "price" or any cohort containing "financial" can hit them).
  4. bid_value_unit = unspecified on a row that has a bid_value. Two rows in this snapshot: saks:110 (Company H, $2.6 billion). Confirm against the filing page whether the figure is enterprise or equity value.
  5. row_unit = cohort rows. These represent unnamed counts. Cross-check the count and class against the sentence quoted — especially when a cohort spans multiple classes (the providence-worcester:140 cohort row sets bidder_class=unknown for that reason; you may want a finer attribution).
  6. consortium family rows. The filing's bidding unit drives actor identity. If a member appears as its own bidder elsewhere in the CSV without a clear filing basis for separate bidding conduct, that is a calibration finding.
  7. merger_agreement_executed not the last event for a deal. The standard story is "agreement → optional go-shop → done". If a bid or contact event appears after merger_agreement_executed, the projection priority probably failed in an unusual filing structure; worth a look.
  8. evidence_page mismatch with the deal's Background section. The Background section spans a known page range per filing (for example, mac-gray pp. 33–47). If a row cites a page well outside the section, the citation unit binding chose a wrong paragraph.

What this CSV is not

  • Not the canonical truth. The canonical graph for any deal is output/extractions/{slug}.json and the run-specific output/audit/{slug}/runs/{run_id}/deal_graph.duckdb. The CSV is a projection of those.
  • Not a one-row-per-bidder ledger. This shape was deliberately retired (see commit 3e2f40f Purge stale extraction contracts). Old workflows that expected a fixed bidder-row schema will not work.
  • Not a price-projection sheet. The Python-owned post-review fields (bI, bI_lo, bI_hi, bF, admitted, formal_boundary, dropout_mechanism, T) are not in this CSV. They are reserved for downstream econometric tooling.
  • Not Alex's 2026 workbook. The legacy workbook is calibration material. When the filing and the workbook disagree, the filing wins. (AGENTS.md, "Source of Truth" section.)

AI-assisted review path

The CSV is the lowest-friction view, but for structural questions — "every dropout in petsmart-inc with no specified reason", "all financial sponsors who later pulled financing for a strategic" — a spreadsheet is awkward. The pipeline produces two richer artefacts that are designed for AI-mediated review.

Artefact paths (referenced in the diagram):

  • JSONL review rows → output/review_rows/{slug}.jsonl
  • DuckDB graph file → output/audit/{slug}/runs/{run_id}/deal_graph.duckdb
  • pages.json → data/filings/{slug}/pages.json

Option 1: Claude Desktop on the JSONL

The simplest AI path: drop output/review_rows/{slug}.jsonl (or several together) into a Claude Desktop conversation. The JSONL files have richer fields than the CSV — every review row carries bound_source_quote, bound_source_page, supplied_quote (the LLM's verbatim quote), issue_codes, suggested_action, the canonical claim ID, the deal/run pair, and so on.

Sample prompts that work well:

  • "In petsmart-inc, list every bid_submitted with its bound page and bid_value. Group by bidder."
  • "For mac-gray, show the dropout chain in chronological order with one line per dropout: who, when, why."
  • "Across all 9 reference deals, find every row where confidence=medium and explain why the quote does not tightly support the typed fields."
  • "For zep, list every actor relation involving New Mountain Capital and its objects."

Option 2: Claude Code on the repository

If you can spare 5 minutes for setup, point Claude Code (the CLI equivalent) at the repo root. It can read every artefact — pages.json, deal_graph_v2.json, state/progress.json, state/flags.jsonl — and connect them to each other. It can also run shell commands, which means it can query DuckDB directly.

Sample prompts:

  • "Open output/audit/mac-gray/runs/{latest}/deal_graph.duckdb and list every actor with bidder_class=unknown plus the events involving each."
  • "Compare the event_subtype distribution across the three saks runs in the stability proof; tell me what changed."
  • "For petsmart-inc, find the Background section of pages.json, pull out the paragraphs cited by evidence_page 32–34, and check whether the bound quotes are accurate."

Option 3: DuckDB CLI directly

For users comfortable with SQL, the per-run DuckDB file is queryable without any AI:

duckdb output/audit/petsmart-inc/runs/{run_id}/deal_graph.duckdb

Sample queries:

-- All bidders and their classes for one deal:
SELECT actor_label, actor_kind, bidder_class FROM actors;

-- All events in chronological order with the bidding actor:
SELECT e.event_subtype, e.event_date, l.actor_label
FROM events e LEFT JOIN event_actor_links l USING (event_id)
ORDER BY e.event_date NULLS LAST;

-- Participation counts by stage:
SELECT process_stage, actor_class, count_min, count_max, count_qualifier
FROM participation_counts;

-- Claims with a non-supported disposition (review-only):
SELECT c.claim_id, c.claim_type, d.disposition, d.reason
FROM claims c JOIN claim_dispositions d USING (claim_id)
WHERE d.disposition <> 'supported' AND d.current = true;

The schema is in pipeline/deal_graph/store.py and contains roughly 25 tables — actors, actor_relations, events, bids (as events), participation_counts, claims, claim_dispositions, claim_evidence, evidence, row_evidence, event_actor_links, plus a few bookkeeping tables for coverage obligations.

Which to use when

Question Recommended artefact
