# Robust Extraction Redesign

Date: 2026-04-29

Status: design approved in chat; implementation plan not yet written.

## Goal

Make the SEC Background-of-the-Merger extraction pipeline robust against run-to-run
model variation by changing the internal contract from direct final-row emission
to paragraph-grounded candidate facts plus deterministic Python compilation.

This is a breaking redesign. The repo's no-backward-compatibility rule applies:
when the schema, prompt contract, state format, or output format changes, update
the live contracts, regenerate affected artifacts, and delete stale paths. Git
history is the compatibility record.

## Problem Statement

The current extractor asks one high-reasoning model call to produce the final
spreadsheet-shaped extraction: event rows, bidder identity, row order, dates,
evidence quotes, lifecycle closure, flags, and several derived status fields.
That makes the model responsible for both reading and bookkeeping.

The high-vs-xhigh reference runs showed material instability:

- Anonymous bidder and cohort accounting changed substantially.
- Date uncertainty flags substituted for each other across runs.
- Final-round decomposition changed enough to create or remove hard flags.
- Formal-stage enrichment fields moved between inferred, inconsistent, and null
  states.
- Bid-value representation and free-text bid-type notes changed phrasing and
  sometimes field placement.

The instability is not solved by higher reasoning effort. It reflects an
under-specified intermediate representation and too much deterministic work
living in the prompt.

## Design Principles

1. The SEC filing remains ground truth.
2. Alex's workbook remains calibration data, not an oracle.
3. Models should read and classify evidence; Python should perform mechanical
   compilation.
4. Every emitted row must remain evidence-backed.
5. Hidden semantics should become structured fields, not flag prose.
6. The pipeline should fail loudly on stale or malformed inputs.
7. Do not restore old workbook labels unless they encode a research distinction
   that the new taxonomy cannot represent.
8. Do not add broad fallback agents or JSON repair calls that hide the original
   failure.

## Proposed Architecture

The durable internal unit should become a paragraph-grounded candidate fact,
not a final event row.

```text
SEC filing
  -> deterministic Background section segmentation
  -> paragraph/page indexed Background text
  -> model extracts paragraph-backed candidate facts
  -> Python canonicalizes dates, ordering, bidder handles, quotes, and counts
  -> narrow semantic classifiers resolve bounded ambiguous fields
  -> Python emits final schema rows
  -> stability harness compares repeated runs
```

Candidate facts should look like this conceptually:

```json
{
  "candidate_id": "cand_0037",
  "paragraph_ids": ["p034_006"],
  "raw_date_text": "On April 14, 2014",
  "actor_text": "five parties, comprising four financial buyers and one strategic buyer",
  "action_text": "submitted preliminary and non-binding indications of interest",
  "candidate_kind": "bid_submission",
  "raw_price_text": "between $20.00 and $23.00 per share",
  "raw_stage_text": "preliminary and non-binding",
  "notes": []
}
```

The final output remains row-per-event JSON, but it is generated by a compiler
from candidate facts and scoped classifier results.

## Component Boundaries

### Background Indexer

The Background indexer owns deterministic filing preparation:

- isolate the Background section;
- preserve original page numbers;
- split text into stable paragraph IDs;
- store paragraph text, page number, order, and optional character offsets;
- reject missing or obviously bad section slices before any model call.

Paragraph IDs are the evidence spine for the rest of the pipeline.

### Candidate Extractor

The candidate extractor is a model call that emits candidate facts, not final
rows. It should identify evidence-backed candidate events and raw text spans:

- event-like action;
- actor or bidder text;
- date phrase;
- price or consideration phrase;
- process-stage phrase;
- paragraph IDs;
- short ambiguity notes only where needed.

It should not assign final `BidderID`, final `source_quote`, final
`source_page`, final placeholder names, final row order, or lifecycle closure.

### Canonicalizer

The canonicalizer is Python-owned. It compiles candidates into normalized
entities and final rows:

- stable event ordering;
- `BidderID` assignment;
- `source_quote` and `source_page` materialization from paragraph IDs;
- exact-count anonymous placeholder expansion;
- bidder placeholder naming;
- bidder registry construction;
- date normalization from raw date phrases;
- rough-date midpoint rules;
- `auction` calculation;
- lifecycle checks and generated `DropSilent` rows;
- row-type required field enforcement;
- final output schema emission.

### Scoped Classifiers

Scoped classifiers are small model calls used only for bounded semantic
judgments that Python cannot safely infer. Examples:

- `NDA` vs `ConsortiumCA` vs skipped rollover/shareholder CA;
- bidder identity linking from filing language;
- operating acquirer vs legal shell;
- bidder type `s` vs `f`;
- `bid_type` and `bid_type_basis`;
- drop initiator and reason;
- final-round role;
- stale prior process vs current-process prelude.

Each classifier should receive only the relevant paragraphs, candidate facts,
and local context. Broad full-event-list rewrites are out of scope.

### Validator And Stability Harness

The validator continues to enforce hard contract invariants, but it should
distinguish structural failures from diagnostic uncertainty.

The stability harness compares repeated runs by stable identities:

- paragraph coverage;
- candidate count by candidate kind;
- final row count by `bid_note`;
- bidder/cohort lifecycle conservation;
- hard flag deltas;
- taxonomy-sensitive soft flag deltas;
- final-round pairing metrics;
- formal-stage transition matrix;
- date-basis distribution;
- bid-value representation states.

Raw total flag count should not be the primary stability metric because xhigh
can be more verbose in info flags without being substantively worse.

## Provider And Linkflow Contract

The design must respect `docs/linkflow-extraction-guide.md`.

For Linkflow, the LLM path must use:

- direct `AsyncOpenAI` client;
- Responses streaming endpoint;
- prompt-only JSON;
- Background-section slice only;
- Python schema and contract checks after parsing.

The implementation must not rely on strict OpenAI Structured Outputs,
`json_schema`, or `text.format` payloads when `OPENAI_BASE_URL` points at
Linkflow/NewAPI-compatible providers. Those payloads are known to be brittle
through Linkflow and can break otherwise valid extraction calls.

Structured Outputs remain a useful conceptual target for native OpenAI
providers, but the Linkflow path is prompt-only JSON plus local validation. This
is not a relaxed contract: stale fields, missing required fields, malformed JSON,
invalid vocabularies, and evidence failures must hard-fail locally.

The implementation should also preserve the Linkflow guide's negative rules:

- no non-streaming Chat Completions for large high-reasoning extraction prompts;
- no runtime schema probes before the real extraction;
- no JSON repair model calls after malformed output;
- no full-filing prompts;
- no low extractor `max_output_tokens` caps that create truncated JSON;
- no per-deal token-budget cap that hides difficult cases;
- no xhigh worker pools above the tested provider ceiling;
- no cached raw-response reuse after prompt, schema, rulebook, or section-slice
  changes.

## Taxonomy Decisions

The current 18-value `bid_note` vocabulary should mostly remain. It preserves
Alex's important concepts while removing overloaded workbook labels.

Keep:

- `Bidder Interest`
- `Bidder Sale`
- `Target Sale`
- `Target Sale Public`
- `Activist Sale`
- `Press Release`
- `IB`
- `IB Terminated`
- `NDA`
- `ConsortiumCA`
- `Drop`
- `DropSilent`
- `Bid`
- `Final Round`
- `Auction Closed`
- `Executed`
- `Terminated`
- `Restarted`

Do not restore shattered workbook labels such as `Final Round Inf Ext Ann` as
top-level event codes. Represent those distinctions as structured axes.

### Add `date_basis`

Replace overlapping date uncertainty flags with a row-level basis enum:

- `precise_text`
- `rough_text_collapsed`
- `context_inferred`
- `sequence_only`
- `unmapped_phrase`
- `unknown`

The existing date fields still carry normalized values. The basis explains how
the date was obtained.

### Add `bid_type_basis`

Replace free-text `bid_type_inference_note` as the main code with:

- `range_bid`
- `nonbinding_ioi`
- `pre_nda_price_signal`
- `loi_subject_to_diligence`
- `final_bid_deadline`
- `best_and_final`
- `merger_agreement_markup`
- `financing_commitment`
- `process_position_fallback`
- `ambiguous`

An optional short human note may remain, but stability should compare the enum,
not prose.

### Replace Formal-Stage Booleans With State

Replace or compile from `invited_to_formal_round` and
`submitted_formal_bid` into:

- `not_invited`
- `invited_no_bid`
- `submitted_bid`
- `unknown`

This better matches the research question and avoids unstable combinations of
two independent booleans.

### Add `final_round_role`

Represent final-round event structure as:

- `announcement`
- `deadline_or_submission_milestone`
- `extension`
- `announcement_and_deadline_same_day`

Keep `final_round_informal` as a separate axis. This directly addresses
same-day announcement/deadline cases such as the Mac Gray instability.

### Add Anonymous Cohort Fields

Add first-class cohort structure for unnamed bidder groups:

- `cohort_id`;
- `cohort_size`;
- `cohort_basis` with values such as `exact_count`, `vague_plural`, and
  `named_subset`.

The compiler should use cohorts to expand exact counts and avoid double
counting named parties that are included inside a larger anonymous group.

### Add Drop Group Basis

Keep `drop_initiator` and `drop_reason_class`, but add:

- `drop_group_basis = named_list | exact_count | vague_group | null`

This stabilizes group drop allocation and avoids burying group logic in
`additional_note` or flags.

### Resolve Dropout Provenance

Before implementation changes, resolve the mapping between Alex's original
dropout labels and the current taxonomy:

- workbook `DropM`;
- PDF `DropBelowM`;
- `DropBelowInf`;
- `DropAtInf`;
- `DropTarget`.

The current converter mapping does not cleanly match Alex's prose. This should
be written as an explicit rulebook decision before regenerating references.

### Demote Or Remove Low-Signal Fields

`comments` is currently unused and should be removed from the model-facing
contract unless a downstream projection requires a null column.

`press_release_subject = "other"` should not be a stability sentinel. It should
remain narrow or be removed.

Legal counsel fields may remain because Alex flagged them as potentially useful,
but they are peripheral to informal bidding and not tested by Alex reference
JSONs. If kept, they should have explicit deal-level evidence.

## What Moves To Python

Python should own:

- paragraph indexing;
- source quote and page generation;
- quote length enforcement;
- final row order;
- `BidderID`;
- bidder placeholder names;
- exact-count anonymous expansion;
- lifecycle closure and `DropSilent`;
- `auction`;
- date normalization;
- `date_basis`;
- rough-date rules;
- final schema projection;
- hard invariant checking;
- stability metrics;
- stale cache rejection.

## What Stays Model-Owned

Models should still own:

- candidate event detection;
- semantic event classification;
- entity clustering from ambiguous filing language;
- CA type classification;
- bidder type classification;
- bid formal/informal classification basis;
- drop agency and reason;
- final-round role when prose is ambiguous;
- process-boundary judgments for stale prior attempts and restarts;
- buyer-group constituent recognition.

## Phased Delivery

### Phase 0: Immutable Run Archive

Stop overwriting comparable high/xhigh artifacts. Every fresh run should have a
durable run ID with raw responses, parsed output, audit metadata, model/provider
settings, rulebook hashes, prompt hashes, and validation reports.

### Phase 1: Background Paragraph Index

Implement deterministic Background section slicing and paragraph/page indexing.
Add fixtures and tests before changing extraction semantics.

### Phase 2: Candidate-Fact Extraction

Introduce the model-facing candidate schema. Run it on the nine reference deals
and inspect paragraph coverage before compiling final rows.

### Phase 3: Deterministic Compiler

Compile candidates into final rows, bidder registry, dates, quotes, pages,
cohorts, and lifecycle rows. Existing final schema fields may be replaced where
the new taxonomy is cleaner.

### Phase 4: Scoped Semantic Classifiers

Add narrow classifiers only for fields that cannot be deterministic. Each
classifier should have a bounded input shape and targeted tests.

### Phase 5: Stability Gate

Run repeated reference-set extractions under the approved Linkflow settings.
Gate target-deal rollout on stable paragraph coverage, zero unexpected hard
flags, stable bidder/cohort lifecycle counts, stable final-round metrics, and
Austin's manual verification against filings.

## Explicit Non-Goals

- Do not run the 392 target deals during this redesign.
- Do not add backward-compatible readers for old extraction artifacts.
- Do not restore the old workbook-shaped schema.
- Do not add broad second-pass full-output rewrite agents.
- Do not use model-based JSON repair.
- Do not make OpenAI Structured Outputs required for Linkflow.
- Do not gate shipping on F1 against Alex's workbook alone.
- Do not treat generated flags as a substitute for structured variables.

## Acceptance Criteria For The Later Implementation

The implementation plan should be considered complete only when:

1. Linkflow calls still use prompt-only JSON over Responses streaming.
2. No strict structured-output payload is sent to Linkflow.
3. Background paragraph IDs are stable and tested.
4. Candidate facts preserve paragraph evidence.
5. Python emits final `source_quote` and `source_page`.
6. Exact-count anonymous cohorts compile deterministically.
7. Bidder lifecycle accounting is stable across repeated runs.
8. Final-round role and formal-stage state no longer depend on free-form notes.
9. Date uncertainty is represented by `date_basis`.
10. Dropout legacy mappings are resolved in the rulebook.
11. Repeated reference-set runs are archived instead of overwritten.
12. The target-deal gate remains closed until Austin verifies all nine reference
    deals and the rulebook is stable across the required clean runs.
